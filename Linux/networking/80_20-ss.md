`ss` is what `netstat` should have been: faster, cleaner, and closer to the kernel’s actual socket state. The 80/20 is about **querying sockets with the right filters and decoding just enough output to act**.

---

# The 80/20 Guide to ss

## Mental Model (core idea)

`ss` = **direct view into kernel socket tables**

It answers:

* What sockets exist?
* Which are listening vs connected?
* Which process owns them?

Think: `netstat`, but **zero lag + better filtering**.

---

## The One Command to Remember

```bash id="kq2k7v"
ss -tulnp
```

Same muscle memory as `netstat`, but faster.

### Flags:

* `-t` → TCP
* `-u` → UDP
* `-l` → listening sockets
* `-n` → numeric (no DNS)
* `-p` → process info

This is your “what’s running on this machine?” command.

---

## The 3 Core Use Cases

## 1. What services are listening?

```bash id="km7v6v"
ss -tulnp
```

Example:

```id="4o2v3l"
LISTEN 0 128 0.0.0.0:80  0.0.0.0:*  users:(("nginx",pid=1234))
```

Interpretation:

* `0.0.0.0:80` → all interfaces
* `LISTEN` → server socket
* `nginx` → owning process

---

## 2. What connections are active?

```bash id="7a7p7t"
ss -tn
```

Shows:

* Established connections
* Source ↔ destination pairs

---

## 3. Who owns a port?

```bash id="8ng7qp"
ss -tulnp | grep :3000
```

Same workflow as `netstat`, just faster.

---

## The Feature That Changes Everything: Filters

This is where `ss` pulls ahead.

---

### Filter by port

```bash id="tq46zb"
ss -tn '( sport = :443 )'
```

---

### Filter by destination

```bash id="h7m3ci"
ss -tn '( dport = :443 )'
```

---

### Combined filter

```bash id="0q9b89"
ss -tn '( sport = :22 or dport = :22 )'
```

This avoids piping to `grep` and is much more precise.

---

## Minimal Output Decoding

Columns:

* **State** → LISTEN / ESTAB / TIME-WAIT
* **Local Address:Port**
* **Peer Address:Port**
* **users:(...)** → process info

---

## The Only States That Matter

* `LISTEN` → waiting for connections
* `ESTAB` → active connection
* `TIME-WAIT` → normal cleanup phase

---

## High-Value Recipes

### Check if a service is running

```bash id="knlm92"
ss -tulnp | grep :80
```

---

### Debug “port already in use”

```bash id="sh1u1g"
ss -tulnp | grep :PORT
```

---

### See all outbound HTTPS connections

```bash id="g3i4u0"
ss -tn '( dport = :443 )'
```

---

### Count active connections

```bash id="6b8y1s"
ss -tn | wc -l
```

---

### Show only established connections

```bash id="xx2q6w"
ss -tn state established
```

---

## Performance Insight (why `ss` is better)

Unlike netstat:

* `ss` reads directly from kernel structures
* Avoids heavy `/proc` parsing
* Scales better under high connection counts

This matters on:

* servers
* containers
* high-traffic systems

---

## Pitfalls to Avoid

### 1. Forgetting `-n`

Same issue as before—DNS slows everything down.

---

### 2. Missing `-p`

No process visibility → harder debugging.

---

### 3. Not using filters

If you rely on `grep`, you’re underusing `ss`.

Filters are:

* faster
* more precise
* composable

---

## Minimal Cheat Sheet

```bash id="m3uk1o"
# listening ports + processes
ss -tulnp

# active connections
ss -tn

# find port owner
ss -tulnp | grep :PORT

# filter by port
ss -tn '( dport = :443 )'

# established only
ss -tn state established

# full visibility
sudo ss -tulnp
```

---

## Final Takeaway

You don’t “learn `ss`”—you internalize three moves:

* Inspect listeners → `-tulnp`
* Inspect connections → `-tn`
* Filter precisely → `( dport / sport )`

Everything else is refinement.
