# The 80/20 Guide to lsof

## Mental Model (core idea)

`lsof` = **everything is a file**

On Unix systems:

* sockets = files
* processes = hold file descriptors
* ports = special files

So `lsof` answers:

> *“Which process is using this thing?”*

---

## The One Command to Remember

```bash id="w4z6j9"
lsof -i :PORT
```

Example:

```bash id="z9b3ra"
lsof -i :3000
```

This instantly tells you:

* which process
* which PID
* which user
  …is using a port

---

## The 3 Core Use Cases

## 1. “What’s using this port?”

```bash id="3w3kmp"
lsof -i :8080
```

Output:

```id="p8yy1k"
node  1234 user  ...  TCP *:8080 (LISTEN)
```

This is the **fastest way to debug port conflicts**.

---

## 2. “What network connections exist?”

```bash id="qjz3dr"
lsof -i
```

Shows:

* all TCP/UDP connections
* associated processes

---

## 3. “What files is this process using?”

```bash id="v7p6yy"
lsof -p 1234
```

Useful for:

* debugging leaks
* seeing open files, sockets, logs

---

## The 3 Flags That Matter Most

### 1. `-i` (network focus)

```bash id="d6gh9g"
lsof -i
```

Filters to:

* sockets only (not regular files)

---

### 2. `-p` (process focus)

```bash id="e3fy7g"
lsof -p PID
```

Inspect everything a process touches.

---

### 3. `-u` (user filter)

```bash id="6n0yfm"
lsof -u username
```

See all files/process resources for a user.

---

## High-Value Patterns

### Kill process using a port

```bash id="rq9sdl"
lsof -i :3000
kill -9 <PID>
```

---

### Only listening sockets

```bash id="fwd7um"
lsof -iTCP -sTCP:LISTEN
```

---

### Filter by protocol

```bash id="yq1p2c"
lsof -iTCP
lsof -iUDP
```

---

### Find deleted but still open files (important!)

```bash id="r9y9s2"
lsof | grep deleted
```

This catches:

* disk space leaks
* processes holding removed files

---

## Output Decoding (minimal)

Columns:

* **COMMAND** → process name
* **PID** → process ID
* **USER** → owner
* **FD** → file descriptor
* **TYPE** → file/socket type
* **NAME** → file path or network endpoint

---

### Example:

```id="zqpxlb"
python 1234 user ... TCP localhost:5000->remote:443 (ESTABLISHED)
```

Interpretation:

* Python process
* active HTTPS connection

---

## Why `lsof` is Powerful

Compared to tools like:

* ss
* netstat

`lsof` uniquely answers:

> **“Which process owns this resource?”**

Not just “what exists,” but **ownership**.

---

## Pitfalls to Avoid

### 1. Forgetting sudo

```bash id="5n0q3v"
sudo lsof -i
```

Without it:

* incomplete results
* missing processes

---

### 2. Too much output

```bash id="8fw9tp"
lsof
```

This dumps *everything* → overwhelming.

Always filter (`-i`, `-p`, `-u`).

---

### 3. Misreading FD column

FD types:

* `cwd` → current working directory
* `txt` → executable
* `mem` → memory-mapped file

Not just “files.”

---

## Minimal Cheat Sheet

```bash id="7j94j8"
# process using port
lsof -i :PORT

# all network connections
lsof -i

# process file usage
lsof -p PID

# user activity
lsof -u username

# listening sockets
lsof -iTCP -sTCP:LISTEN

# full visibility
sudo lsof -i
```

---

## Final Takeaway

You don’t need most of `lsof`.

If you internalize:

* `lsof -i :PORT` → who owns this port
* `lsof -p PID` → what is this process doing
* `lsof -i` → what network activity exists

…you unlock its real value.
