Here’s a tight **80/20 guide to SSH**—focused on the small set of concepts and configurations that deliver most of the real-world value.

---

# 1) What SSH actually is (in practical terms)

**OpenSSH** is the de facto standard implementation.

At its core, SSH gives you:

* **Encrypted remote shell access**
* **Secure file transfer** (via `scp` / `sftp`)
* **Tunneling** (forwarding ports securely)

Think: *secure pipe + remote terminal + authentication layer.*

---

# 2) The 80/20 usage commands

### Connect to a server

```bash
ssh user@host
```

With a key:

```bash
ssh -i ~/.ssh/id_ed25519 user@host
```

Specify port:

```bash
ssh -p 2222 user@host
```

---

### Copy files

```bash
scp file.txt user@host:/path/
scp -r folder/ user@host:/path/
```

---

### Open an interactive file transfer session

```bash
sftp user@host
```

---

### Port forwarding (high leverage)

Local → remote:

```bash
ssh -L 8080:localhost:3000 user@host
```

Now `localhost:8080` maps to remote `localhost:3000`.

---

# 3) Key-based authentication (non-negotiable)

Passwords are the weak link. Use keys.

### Generate a key

```bash
ssh-keygen -t ed25519 -C "your_email@example.com"
```

Why **ed25519**?

* Faster
* Stronger per bit than RSA
* Smaller keys

---

### Copy key to server

```
ssh-copy-id -i ~/.ssh/filename.pub user@192.168.1.145
```

Manual fallback:

```bash
cat ~/.ssh/id_ed25519.pub | ssh user@host "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"
```

---

# 4) SSH config file (massive productivity boost)

Edit:

```bash
~/.ssh/config
```

Example:

```bash
Host myserver
    HostName example.com
    User ubuntu
    Port 2222
    IdentityFile ~/.ssh/id_ed25519
```

Now:

```bash
ssh myserver
```

---

# 5) Critical security hardening (the real 20%)

On the server, edit:

```bash
/etc/ssh/sshd_config
```

---

## 🔒 Minimum viable secure setup

### 1. Disable password auth

```bash
PasswordAuthentication no
```

### 2. Disable root login

```bash
PermitRootLogin no
```

### 3. Use a non-default port (optional but useful)

```bash
Port 2222
```

### 4. Restrict users

```bash
AllowUsers youruser
```

### 5. Use modern keys only

```bash
PubkeyAuthentication yes
```

---

Restart SSH:

```bash
sudo systemctl restart ssh
```

---

# 6) File permissions (often overlooked)

SSH is strict:

```bash
chmod 700 ~/.ssh
chmod 600 ~/.ssh/authorized_keys
chmod 600 ~/.ssh/id_ed25519
```

If permissions are too open → SSH refuses to work.

---

# 7) Defense-in-depth (high ROI extras)

### Fail2Ban (brute-force protection)

Fail2Ban blocks repeated login attempts.

```bash
sudo apt install fail2ban
```

---

### Firewall (UFW example)

```bash
sudo ufw allow 2222/tcp
sudo ufw enable
```

---

### SSH agent (for key management)

```bash
eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
```

---

# 8) Common mistakes to avoid

* Leaving password login enabled
* Using RSA keys when ed25519 is available
* Forgetting to test access **before closing current session**
* Not backing up your private key
* Misconfigured permissions breaking login

---

# 9) Mental model (so it sticks)

SSH has 3 layers:

1. **Transport encryption** → secure channel
2. **Authentication** → keys vs password
3. **Session** → shell, file transfer, or tunnel

If something breaks, it's almost always one of those.

---

# 10) Minimal secure workflow (what you should actually do)

1. Generate key (`ed25519`)
2. Copy to server
3. Disable password auth
4. Disable root login
5. Add SSH config alias
6. Install Fail2Ban

That’s ~80% of real-world SSH competence.
