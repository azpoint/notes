`wget` is less about “downloading files” and more about **reliable, scriptable data retrieval over HTTP/FTP**. The 80/20 is: grab files, mirror content, and make downloads resilient.

---

# The 80/20 Guide to wget

## Mental Model (core idea)

`wget` = **non-interactive downloader**

It’s designed for:

* scripts
* cron jobs
* unstable networks

Think: `curl` without interactivity, but with **persistence + recursion**.

---

## The One Command to Remember

```bash id="b4r7pn"
wget URL
```

Example:

```bash id="wcz9ls"
wget https://example.com/file.zip
```

Downloads the file into your current directory.

---

## The 3 Core Use Cases

## 1. Download a file

```bash id="f1k3va"
wget https://example.com/file.tar.gz
```

That’s the baseline.

---

## 2. Resume a broken download

```bash id="3zux1v"
wget -c https://example.com/large.iso
```

* `-c` → continue
* Critical for large files or flaky connections

---

## 3. Download and rename

```bash id="1o3z7p"
wget -O myfile.zip https://example.com/file.zip
```

* `-O` → output filename

---

## The 3 Flags That Matter Most

### 1. `-c` (continue)

```bash id="gwdwqf"
wget -c URL
```

Prevents restarting large downloads from scratch.

---

### 2. `-O` (output file)

```bash id="ezs8b2"
wget -O name.ext URL
```

Essential for scripting.

---

### 3. `-q` (quiet)

```bash id="y2u4s7"
wget -q URL
```

Suppresses output—useful in automation.

---

## High-Value Patterns

### Download in background

```bash id="4wrs9f"
wget -b URL
```

* `-b` → background
* logs to `wget-log`

---

### Limit download speed

```bash id="pxq0yy"
wget --limit-rate=200k URL
```

Prevents bandwidth saturation.

---

### Retry automatically

```bash id="7n0gqg"
wget --tries=10 URL
```

Handles unstable connections gracefully.

---

### Timeout control

```bash id="6a9n2s"
wget --timeout=10 URL
```

Avoids hanging forever.

---

## Recursive Downloads (where wget shines)

### Mirror a website

```bash id="4a6vhm"
wget -r https://example.com
```

* `-r` → recursive

---

### Download with depth limit

```bash id="w7uv0h"
wget -r -l 2 https://example.com
```

* `-l` → recursion depth

---

### Mirror properly (usable offline)

```bash id="j3v8yq"
wget -r -k -p https://example.com
```

* `-k` → convert links for local use
* `-p` → download page assets (CSS, images)

---

## Authentication (basic use)

```bash id="yt6cbb"
wget --user=username --password=password URL
```

---

## Headers / API usage (lightweight)

```bash id="y0r3my"
wget --header="Authorization: Bearer TOKEN" URL
```

Useful for simple API pulls when you don’t need full curl flexibility.

---

## Output Insight

Typical output:

```id="a7yzzp"
Saving to: ‘file.zip’
```

Key signals:

* progress bar → active download
* `saved` → success
* retry messages → network issues handled

---

## Performance Insight

`wget` is optimized for:

* **resumability**
* **retry logic**
* **long-running transfers**

This makes it superior to naive download methods in:

* CI pipelines
* remote servers
* unreliable networks

---

## Pitfalls to Avoid

### 1. Overwriting files unintentionally

Use:

```bash id="b2y5o6"
wget -nc URL
```

* `-nc` → no clobber (don’t overwrite)

---

### 2. Infinite recursion

```bash id="x9p2uh"
wget -r URL
```

can explode in scope.

Always constrain:

```bash id="9ytw0v"
wget -r -l 2 URL
```

---

### 3. Misusing for APIs

`wget` works, but for complex requests:
→ use curl

---

## wget vs curl (quick 80/20)

* `wget` → downloading, mirroring, automation
* `curl` → APIs, debugging requests, fine control

---

## Minimal Cheat Sheet

```bash id="ybdk6p"
# download file
wget URL

# resume
wget -c URL

# rename
wget -O file URL

# quiet mode
wget -q URL

# recursive
wget -r -l 2 URL

# mirror site
wget -r -k -p URL

# no overwrite
wget -nc URL
```

---

## Final Takeaway

You don’t need most of `wget`.

If you understand:

* basic download (`wget URL`)
* resume (`-c`)
* rename (`-O`)
* recursion (`-r`, `-l`)

…you can handle almost every real-world use case.
